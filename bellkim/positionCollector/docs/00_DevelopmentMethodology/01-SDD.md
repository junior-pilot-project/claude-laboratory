# 01 · SDD (Spec Driven Development)

## SDD란?

> "소프트웨어를 어떻게 만들 것인지를 설명하는 설계 문서"

요구사항(WHAT)이 아닌 구현 방법(HOW)에 집중하며, IEEE 1016 표준을 기반으로 한다.

SDD(Spec Driven Development)는 코딩을 시작하기 전에 팀 전체가 동의한 아키텍처와 설계를 문서로 확정하는 방법론이다. AI 코딩 에이전트에게 SDD를 컨텍스트로 제공하면 일관되고 예측 가능한 결과물이 나온다.

## 왜 SDD가 필요한가?

- AI에게 막연히 "기능 만들어줘"라고 하면 AI는 매번 다른 방향으로 구현한다
- SDD가 있으면 AI는 이미 확정된 설계 안에서 코드를 생성하므로 불필요한 리팩터링이 줄어든다
- 여러 팀원이 AI를 쓸 때 일관된 아키텍처를 유지할 수 있다
- 새로운 팀원 온보딩 시 SDD 하나로 전체 시스템 이해 가능

## SDD 작성 흐름

1. **요구사항 정의** — SRS / 사용자 스토리 작성
2. **아키텍처 설계** — 컴포넌트 분해 및 의존 관계 정리
3. **UI / 데이터 설계** — 화면 흐름 및 데이터 모델 확정
4. **SDD 문서화** — IEEE 1016 포맷으로 작성
5. **AI 코딩 시작** — SDD를 CLAUDE.md 또는 컨텍스트로 제공

## SDD 주요 포함 항목

### 📄 제목 & 저자
문서의 목적, 대상 독자, 버전 정보를 기록한다.

### 🏗️ 시스템 개요
소프트웨어의 목적, 범위, 제약 조건을 기술한다.

### 🧩 아키텍처 설계
서브시스템 분해 및 컴포넌트 관계도를 포함한다. 설계 패턴 결정 이유도 기록.

### 🖥️ 인터페이스 설계
UI 흐름, 외부 시스템 연동 명세, API 계약을 정의한다.

### 🗄️ 데이터 설계
데이터 구조, DB 스키마, 로컬 스토어 설계를 포함한다.

### 🎬 운영 시나리오
사용자 관점의 주요 시나리오를 단계별로 기술한다.

## AI와 함께 SDD 활용하기

```markdown
# CLAUDE.md 예시
이 프로젝트는 SDD 기반으로 개발됩니다.
- 아키텍처: specs/architecture.md 참고
- 데이터 모델: specs/data-model.md 참고
- UI 흐름: specs/ui-flow.md 참고
새 기능 추가 전 반드시 해당 스펙 문서를 먼저 읽으세요.
```

> 💡 **핵심 교훈**: 문서 작성에 쓰는 시간이 나중의 리팩터링 시간을 크게 줄여준다. "문서 쓸 시간 없다"고 하면, 나중에 더 많은 시간을 낭비하게 된다.
